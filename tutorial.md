# Deployment Tutorial — NXP i.MX93 (Debian)

This tutorial walks you through building the server on your laptop and deploying it to an NXP i.MX93 board running Debian. By the end you will have the web dashboard running on the device, reachable from your browser, with live sensor data and ML inference.

---

## What you need

| Item | Notes |
|------|-------|
| NXP i.MX93 board running Debian | Any Debian-based image works |
| Laptop (Linux / WSL2 / macOS) | For cross-compilation |
| Tri-axial RS485 accelerometer | KAX301, KAX302, S6S, 12B, or 15B |
| RS485 ↔ UART adapter | Built-in on some boards; USB dongle otherwise |
| Ethernet / Wi-Fi | Board and laptop must be on the same network |
| eIQ TSS output files | `algorithm.dat`, `libtss_svm.a`, `TimeSeries.h` — generated by NXP eIQ Time Series Studio; see the ML section |

---

> **No code changes?** Skip to [Step 3 — Copy files to the device](#3-copy-files-to-the-device).

## 1. Host setup (laptop)

### 1.1 Install Rust

```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source "$HOME/.cargo/env"
```

### 1.2 Add the aarch64 target

```bash
rustup target add aarch64-unknown-linux-gnu
```

### 1.3 Install the cross-linker

On Debian/Ubuntu or WSL2:
```bash
sudo apt install gcc-aarch64-linux-gnu
```

On macOS:
```bash
brew install aarch64-unknown-linux-gnu
```

Verify it is on your PATH:
```bash
aarch64-linux-gnu-gcc --version
```

---

## 2. Build for aarch64

Clone (or unzip) the codebase, then from the project root:

```bash
cargo build --release --target aarch64-unknown-linux-gnu
```

The binary is at:
```
target/aarch64-unknown-linux-gnu/release/modbus-stream
```

The `.cargo/config.toml` already sets the linker, `target-cpu=cortex-a55`, and `target-feature=+neon` automatically for this target — no extra flags needed.

> **Note:** The build links `lib/libtss_svm.a` (the SVM ML library) only for aarch64. The binary is the only thing that needs to be compiled; the `.a` file is baked in.

### When to skip recompilation

You only need to rebuild the binary when `.rs` source files change. For everything else, copy the updated files directly and restart the server:

| What changed | Action needed |
|--------------|---------------|
| `src/*.rs` | Rebuild + copy binary |
| `lib/libtss_svm.a` | Rebuild + copy binary (library is baked in at link time) |
| `templates/`, `static/` | `scp` the directory, no rebuild |
| `config.toml` | `scp` the file, no rebuild |
| `algorithm.dat` | `scp` the file, restart server |

## 3. Copy files to the device

Find the board's IP address by running this on the board over a serial console or monitor:

```bash
ip addr show
```

Look for an `inet` line under your active interface (e.g. `eth0` or `end0`):

```
2: eth0: ...
    inet 192.168.1.142/24 ...
```

The address is the part before the `/` — in this example `192.168.1.142`.

Run the following on your **laptop** to copy everything in one go (replace the address and path with your own):

```bash
scp -r lib/ templates/ static/ data/ config.toml algorithm.dat \
    target/aarch64-unknown-linux-gnu/release/modbus-stream \
    root@192.168.1.142:/path/to/folder/
```

## 4. Physical wiring

### RS485 signal connections

| Sensor wire | Board / adapter terminal |
|-------------|--------------------------|
| A+ (DATA+)  | A / D+                   |
| B− (DATA−)  | B / D−                   |
| GND         | GND                      |
| VCC         | 5 V or 12 V (check your sensor spec) |

- Keep the cable short or add a 120 Ω termination resistor across A/B at the far end.
- A and B **must not be swapped** — if you get Modbus timeouts, try reversing them.

### Finding the serial port on the board

SSH into the board and list UART devices:

```bash
ls /dev/tty*
```

Common names on i.MX93 Debian:
- `/dev/ttymxc0` — `/dev/ttymxc3` for on-board UARTs
- `/dev/ttyUSB0` for a USB RS485 dongle

To confirm data is arriving on the port:
```bash
sudo stty -F /dev/ttymxc2 3000000 raw
sudo cat /dev/ttymxc2 | xxd | head
```

## 5. Configure

Edit `config.toml` on the board (or edit locally and re-copy):

```toml
[server]
host = "0.0.0.0"
port = 3000

[[sensors]]
device = "/dev/ttymxc2"   # your RS485 port
baud_rate = 3000000       # 3 Mbps for raw streaming; 115200 for metrics only
slave_id = 1
timeout_ms = 5000
retry_attempts = 3

# Add more [[sensors]] blocks for additional sensors (up to 4 total)
# [[sensors]]
# device = "/dev/ttymxc3"
# baud_rate = 3000000
# slave_id = 1
# timeout_ms = 5000
# retry_attempts = 3

[streaming]
max_connections = 10
buffer_size = 1024
metrics_update_rate_hz = 5.0
raw_data_max_samples = 123
websocket_ping_interval_sec = 30

[logging]
level = "info"
format = "pretty"
```

**Serial port permissions** — if the server cannot open the port:
```bash
sudo usermod -aG dialout $USER
# Log out and back in, or: newgrp dialout
```

## 6. First launch

SSH into the board and run from the deploy directory:

```bash
cd /opt/modbus-stream
./modbus-stream
```

At startup the server prints a 4-step diagnostic per sensor:

```
════════════════════════════════════════════════════════════
  Modbus Device Startup Diagnostics
════════════════════════════════════════════════════════════
  [1/4] Checking device path ...        OK
  [2/4] Checking permissions ...        OK (read + write)
  [3/4] Opening serial port ...         OK (12 ms)
  [4/4] Testing Modbus communication ... OK (45 ms)
         Model:    KAX301 | Gain: 4G
  Result: CONNECTED
```

If step 4 fails the server still starts in **offline mode** — the UI works but no live data is shown. See the Troubleshooting section.

Open a browser on your laptop and go to:
```
http://192.168.1.142:3000
```

## 7. Dashboard overview

| Page | What it shows |
|------|---------------|
| `/` | Navigation hub |
| `/settings` | Edit sensor config, test connection |
| `/{n}/view/raw` | Live X/Y/Z waveform for sensor n |
| `/{n}/view/metrics` | Live RMS, Peak, Crest Factor, etc. |
| `/view/csv` | Recorded CSV browser + ML inference |

Sensor numbers match the order of `[[sensors]]` blocks in `config.toml` (first block = sensor 1).

## 8. Recording and ML inference

### Record a burst

From the dashboard click **Record** (or `POST /api/record/start`). The server captures ~10 seconds of data at 7812 Hz from all connected sensors and saves timestamped CSV files to `data/`. Poll `GET /api/record/status` for progress.

CSV format: `x,y,z` header followed by 78 120 rows of g-unit float values.

### Run inference

1. Navigate to `/view/csv`
2. Click a filename in the sidebar to open it — the chart and toolbar appear
3. Click **Infer** in the toolbar
4. The result is displayed as a **Class N** badge; hover it to see per-class probabilities

The model takes the middle 1953 samples from the 78 120-sample file and classifies the vibration into one of 4 classes.

## 9. ML model — using your own

All three model files are generated together by **NXP eIQ Time Series Studio** and must be kept in sync with each other:

| File | Where it lives | Role |
|------|----------------|------|
| `lib/libtss_svm.a` | Project `lib/` directory | Compiled SVM inference engine — baked into the binary at link time |
| `TimeSeries.h` | Project `lib/` directory | C API header — used at compile time to call the library |
| `algorithm.dat` | Working directory on the board (`/opt/modbus-stream/`) | Trained model weights — read from disk at server startup |

> **Important:** `libtss_svm.a` and `algorithm.dat` are generated as a matched pair by eIQ TSS. Do not mix files from different training runs or eIQ TSS versions — the inference engine and the weights must correspond to the same export.

To replace the model with your own:

### 9.1 Collect training data

Record CSV files for each class using the **Record** button. Aim for at least 10–20 recordings per class under representative conditions. The server saves them to `data/` automatically.

### 9.2 Label and train in NXP eIQ Time Series Studio

1. Open **NXP eIQ Time Series Studio** on your PC.
2. Create a new project and import your CSV files.
3. Assign class labels to each file (e.g. Class 1 = idle, Class 2 = fault A, …).
4. Train the SVM classifier. The tool lets you tune window size and feature set.
5. Export the model — eIQ TSS produces three files: `algorithm.dat`, `libtss_svm.a`, and `TimeSeries.h`.

### 9.3 Deploy the new model

Always treat the three eIQ TSS output files as a unit. Replace all of them together.

**Step 1 — update the library and header on your laptop, then rebuild:**
```bash
cp /path/to/export/libtss_svm.a lib/
cp /path/to/export/TimeSeries.h  lib/
cargo build --release --target aarch64-unknown-linux-gnu
```

**Step 2 — copy everything to the board:**
```bash
scp target/aarch64-unknown-linux-gnu/release/modbus-stream user@192.168.1.142:/opt/modbus-stream/
scp /path/to/export/algorithm.dat                           user@192.168.1.142:/opt/modbus-stream/
```

**Step 3 — restart the server:**
```bash
# If running manually: Ctrl-C, then ./modbus-stream
# If running as a service:
sudo systemctl restart modbus-stream
```

> If only `algorithm.dat` changed but `libtss_svm.a` did not (e.g. you retrained without updating eIQ TSS), you can skip the rebuild: just `scp algorithm.dat` and restart.

### 9.4 Verify the model loaded

Check the server log at startup:
```
TSS ML model ready: data_tab=0 (interleaved), data_len=1953, data_dim=3, classes=4
```

- `data_tab` — input layout (`0` = interleaved `[x,y,z,x,y,z,…]`, `1` = channels-first `[all_x, all_y, all_z]`). The server reads this automatically from the model and reformats the data accordingly.
- `data_len` — window size the model expects. If you trained with a different window size it will be reflected here and the server adjusts automatically.
- `classes` — number of output classes.

If the line is absent, `algorithm.dat` is missing or the model failed to load.

## 10. Run as a systemd service (auto-start on boot)

Create the service file on the board:

```bash
sudo nano /etc/systemd/system/modbus-stream.service
```

Paste:

```ini
[Unit]
Description=Modbus Stream — Accelerometer Web Interface
After=network.target

[Service]
Type=simple
User=<your-username>
WorkingDirectory=/opt/modbus-stream
ExecStart=/opt/modbus-stream/modbus-stream
Restart=on-failure
RestartSec=5
Environment=RUST_LOG=info

[Install]
WantedBy=multi-user.target
```

Replace `<your-username>` with your actual username on the board.

Enable and start:

```bash
sudo systemctl daemon-reload
sudo systemctl enable modbus-stream
sudo systemctl start modbus-stream
```

Check status and logs:

```bash
sudo systemctl status modbus-stream
sudo journalctl -u modbus-stream -f
```

## 11. Troubleshooting

**Step 4 fails — Modbus timeout**
- Verify slave ID and baud rate match what is programmed on the sensor (default: slave 1, 3 Mbps)
- Check A/B wires are not swapped
- Add 120 Ω termination resistor across A/B if the cable is longer than ~1 m
- Test independently: `mbpoll -a 1 -b 3000000 /dev/ttymxc2 -t 3 -r 20`

**Permission denied on serial port**
```bash
sudo usermod -aG dialout $USER
newgrp dialout
```

**Infer returns error**
- Confirm `algorithm.dat` is in `/opt/modbus-stream/` (the server's working directory)
- Check the startup log for `TSS ML model ready:` — if absent, model init failed
- Ensure `algorithm.dat` and `libtss_svm.a` are from the same eIQ TSS export — mixing versions causes silent inference errors
- ML inference only works on the aarch64 binary; it will always error on x86_64

**Port already in use**
```bash
sudo fuser -k 3000/tcp
```

**No data directory — server won't record**
```bash
mkdir -p /opt/modbus-stream/data
```

**Updating the binary after a code change**

On your laptop:
```bash
cargo build --release --target aarch64-unknown-linux-gnu
scp target/aarch64-unknown-linux-gnu/release/modbus-stream user@192.168.1.142:/opt/modbus-stream/
```

If running as a service:
```bash
sudo systemctl restart modbus-stream
```
